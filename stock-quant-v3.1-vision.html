四大图片识别场景

利用 LLM 多模态视觉能力，从交易 APP 截图中提取数据并智能分析

📊

场景 1：K线图形态识别

用户截图同花顺/东方财富等 APP 的 K 线图，AI 识别蜡烛图形态（头肩顶、双底、旗形等）、趋势方向、关键支撑压力位。

输出 → 形态名称 + 趋势判断 + 操作建议 + 关键价位标注
💼

场景 2：持仓/交割单识别

截图持仓列表或交割单，AI 识别股票代码、持仓数量、成本价、盈亏比例等，生成持仓诊断报告。

输出 → 持仓结构化数据 + 风险评估 + 优化建议
📋

场景 3：个股详情页分析

截图个股详情页面（含 PE/PB/换手率/量比等指标），AI 提取全部关键指标并综合分析基本面和技术面。

输出 → 提取指标表格 + 多维度评分 + 同行业对比
📰

场景 4：新闻/研报要点提取

截图财经新闻、券商研报页面，AI 提取核心观点、目标价、评级变动等关键信息，生成简明摘要。

输出 → 关键要点列表 + 情绪判断 + 对相关股票的影响分析

💡 核心技术思路

这个功能不需要本地 OCR 或计算机视觉模型。Claude (Vision)、GPT-4o、DeepSeek-VL 等多模态大模型本身具备强大的图片理解能力——你发一张 K 线图截图给它，它能直接"看懂"蜡烛图形态、文字数据、指标数值。

所以实现路径非常清晰：前端上传图片 → 后端压缩 + 转 Base64 → 连同场景 Prompt 一起发给 LLM Vision API → 流式返回分析结果。完全复用现有 AI 网关，只需扩展支持图片消息格式。

LLM Vision API 多模型支持

👁️

Claude Vision 推荐

claude-sonnet-4-20250514 / claude-opus-4-6

Anthropic Claude 原生支持 Vision，图片直接放 messages 中的 image content block，支持 Base64 和 URL 两种方式。对金融图表的理解能力很强。

对中文财经截图识别准确度高，推荐作为首选视觉模型
👁️

GPT-4o Vision

gpt-4o / gpt-4o-mini

OpenAI GPT-4o 多模态原生支持，图片作为 image_url 类型 content 传入。4o-mini 成本更低但识别能力略弱。

多模态能力成熟稳定，4o-mini 可用于低成本批量场景
👁️

DeepSeek-VL

deepseek-chat (VL 能力)

DeepSeek 的多模态能力在持续增强中，作为低成本备选。API 调用格式与 OpenAI 兼容。

国内直连 + 低成本，适合非关键场景降级使用

图文并排布局，方便用户对照原图验证 AI 分析
// 前端 Canvas 图片压缩核心逻辑 // client/src/utils/imageCompress.js export function compressImage(file, maxSizeMB = 2, maxDimension = 2048) { return new Promise((resolve) => { const reader = new FileReader(); reader.onload = (e) => { const img = new Image(); img.onload = () => { const canvas = document.createElement('canvas'); let { width, height } = img; // 限制最大尺寸 if (width > maxDimension || height > maxDimension) { const ratio = Math.min(maxDimension / width, maxDimension / height); width *= ratio; height *= ratio; } canvas.width = width; canvas.height = height; canvas.getContext('2d').drawImage(img, 0, 0, width, height); // 递减 quality 直到小于 maxSize let quality = 0.9; let blob; const tryCompress = () => { canvas.toBlob((b) => { if (b.size > maxSizeMB * 1024 * 1024 && quality > 0.3) { quality -= 0.1; tryCompress(); } else { resolve(b); } }, 'image/jpeg', quality); }; tryCompress(); }; img.src = e.target.result; }; reader.readAsDataURL(file); }); }

后端处理链路

图片接收 → 压缩 → Base64 → LLM Vision API → SSE 流式返回

// server/routes/ai.js — 新增 Vision 路由 const multer = require('multer'); const sharp = require('sharp'); const upload = multer({ storage: multer.memoryStorage(), // 内存存储，不写磁盘！(省磁盘 + 2G内存注意限制) limits: { fileSize: 10 * 1024 * 1024 }, // 10MB 上限 fileFilter: (req, file, cb) => { const allowed = ['image/jpeg', 'image/png', 'image/webp']; cb(null, allowed.includes(file.mimetype)); } }); // POST /api/ai/vision — 图片分析 router.post('/vision', auth, upload.single('image'), async (req, res) => { const { scene } = req.body; // 'kline' | 'portfolio' | 'detail' | 'news' | 'auto' // 1. sharp 二次压缩 (确保 ≤1MB, 长边 ≤2048px) const compressed = await sharp(req.file.buffer) .resize(2048, 2048, { fit: 'inside', withoutEnlargement: true }) .jpeg({ quality: 80 }) .toBuffer(); // 2. 转 Base64 const base64 = compressed.toString('base64'); const mediaType = 'image/jpeg'; // 3. 获取场景 Prompt const prompt = promptEngine.getVisionPrompt(scene); // 4. 调用 LLM Vision API (通过现有网关) res.setHeader('Content-Type', 'text/event-stream'); res.setHeader('Cache-Control', 'no-cache'); res.setHeader('Connection', 'keep-alive'); const stream = llmGateway.streamVision( 'stock-vision', // 场景名 → 自动路由到最合适的视觉模型 prompt, { base64, mediaType } ); for await (const chunk of stream) { res.write(`data: ${JSON.stringify({ text: chunk })}\n\n`); } res.write('data: [DONE]\n\n'); res.end(); });
// server/services/ai/llmGateway.js — 扩展 Vision 方法 // 在现有 LLMGateway 类中新增: async *streamVision(scene, prompt, image) { const provider = this.getProviderForScene(scene); // 构建含图片的 messages (各 Provider 格式略有不同) const messages = provider.buildVisionMessages(prompt, image); // 流式调用 yield* provider.stream(messages); } // ---- Claude Adapter 中的 Vision 消息格式 ---- buildVisionMessages(prompt, { base64, mediaType }) { return [{ role: 'user', content: [ { type: 'image', source: { type: 'base64', media_type: mediaType, data: base64 } }, { type: 'text', text: prompt } ] }]; } // ---- OpenAI/DeepSeek Adapter 中的 Vision 消息格式 ---- buildVisionMessages(prompt, { base64, mediaType }) { return [{ role: 'user', content: [ { type: 'image_url', image_url: { url: `data:${mediaType};base64,${base64}` } }, { type: 'text', text: prompt } ] }]; }

⚠️ 2G 内存服务器注意事项

图片不落盘 — multer 用 memoryStorage，sharp 处理完后 Buffer 立即释放，不写磁盘文件。分析完成后只存文本结果到 MongoDB，不存原图。

并发限制 — 同一时刻只允许 2 个 Vision 请求并发（一张 2MB 图片 Base64 后约 2.7MB 内存），通过中间件限流。

前端压缩是关键 — 手机拍照原图可能 5-10MB，必须在前端 Canvas 压缩到 2MB 以内再上传，减轻服务器内存压力。

💰 成本控制策略

① 图片压缩是关键 — 图片尺寸直接影响 token 消耗。2048px 长边、80% JPEG quality 是最佳平衡点，既保证 AI 能看清细节，又不会产生过多 image tokens。

② 模型路由降级 — K 线形态和研报摘要可以用 GPT-4o-mini（便宜），持仓识别需要更准确的数据提取用 Claude/GPT-4o。

③ 结果缓存 — 同一张图片（通过 MD5 hash 判断）不重复分析，直接返回缓存结果。

④ 每用户日限额 — Vision 分析比文本分析贵，单独设置每用户每日 Vision 调用次数上限（如 analyst 角色 30 次/日）。
